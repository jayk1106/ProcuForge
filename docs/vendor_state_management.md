# Vendor Agent — State Management Reference

Frontend reference for `procu_forge_vendor`. Covers every session-state key, the
`VendorThreadStatus` lifecycle, valid transitions, the four negotiation price anchors,
the full message-log structure, and a display guide for each status.

---

## 1. State Keys Reference

All keys live in `tool_context.state` / `callback_context.state` (Google ADK
`InMemorySessionService`). One session = one buyer ↔ vendor RFQ thread.

| Key (string) | Constant | Type | Set by | Contents | Frontend relevance |
|---|---|---|---|---|---|
| `vendor_id` | `VENDOR_ID_KEY` | `str` | `_initial_state_from_rfq` (callbacks.py) | The vendor's ID exactly as it appears in the database and in every A2A envelope | Display vendor name (resolve from vendor directory) |
| `rfq_id` | `RFQ_ID_KEY` | `str` | `_initial_state_from_rfq` | UUID generated by the buyer agent; identifies the negotiation thread | Use as primary key to link buyer ↔ vendor views |
| `round` | `ROUND_KEY` | `int` | `before_agent_callback` on each inbound message | Current negotiation round, mirrors the buyer's envelope `round` field | Show negotiation round indicator |
| `product` | `PRODUCT_KEY` | `dict` | Skeleton from `_initial_state_from_rfq`; enriched by `quote/tools.py` | See product sub-object schema below | Core product display card |
| `communication` | `COMMUNICATION_KEY` | `list[dict]` | `_initial_state_from_rfq` (seed with RFQ); appended by `before_agent_callback` each turn | Chronological list of every A2A envelope in the thread (inbound + outbound) | Negotiation timeline / message log |
| `status` | `STATUS_KEY` | `str` (`VendorThreadStatus`) | `set_status()` called from `before_agent_callback` and subagent callbacks | Current thread lifecycle stage | Primary driver for vendor-side UI |
| `last_selling_price` | `LAST_SELLING_PRICE_KEY` | `float` | `get_negotiation_context` in `negotiation/tools.py` (lazy init) | Vendor's minimum acceptable price = 90% of `listed_unit_price` | **Internal only** — do not display; floor enforced server-side |
| `latest_buyer_price` | `LATEST_BUYER_PRICE_KEY` | `float \| null` | `send_response` in `negotiation/tools.py` when `buyer_proposed_price` is provided | The buyer's most recent counter-offer price | Buyer offer bar in negotiation view |
| `latest_offer_price` | `LATEST_OFFER_PRICE_KEY` | `float \| null` | `quote/tools.py` (opening quote); `send_response` on each vendor COUNTER_OFFER/ACCEPT | The vendor's most recent offer price | Vendor offer bar in negotiation view |
| `accepted_price` | `ACCEPTED_PRICE_KEY` | `float \| null` | `negotiation/callbacks.py` when vendor sends ACCEPT outbound | Final agreed unit price | "Agreed price" badge; use as PO unit price reference |
| `po` | `PO_KEY` | `dict \| null` | `before_agent_callback` when `message_type == "PO"` validates and persists payload | Inbound PO from buyer — see PO sub-object schema below | PO detail card |
| `grn` | `GRN_KEY` | `dict \| null` | `before_agent_callback` when `message_type == "GRN_CREATED"` validates and persists | Inbound GRN from buyer — see GRN sub-object schema below | GRN detail card |
| `vendor_is_final` | `VENDOR_IS_FINAL_KEY` | `bool` | `send_response` when `is_final=True` is passed; prevents further vendor counters | Once True, vendor cannot send another COUNTER_OFFER | Show "Best & Final" badge on vendor's last counter |
| `seen_message_ids` | `SEEN_MESSAGE_IDS_KEY` | `list[str]` (max 50) | `_record_message_id` in `before_agent_callback` | Ring-buffer dedup list for idempotency | **Internal only** — not for display |
| `temp:request_body` | *(no constant)* | `dict` | `vendor_server.py` on each HTTP request | The current inbound A2A envelope for this request | **Internal only** — cleared each turn |
| `temp:response_body` | *(no constant)* | `dict \| null` | Tools (quote, negotiation, purchase) write the outbound envelope here | Current outbound A2A envelope, read and sent by `after_agent_callback` | **Internal only** — cleared each turn |

### 1a. `product` sub-object schema

Populated in two stages: the skeleton is set from the RFQ by `_initial_state_from_rfq`,
and the full record is written by `quote/tools.py` after Firestore lookup.

```json
{
  "id":                  "string  — product UUID",
  "sku":                 "string  — vendor's SKU for this product",
  "unit":                "string  — e.g. 'each', 'kg', 'box'",
  "quantity":            "int     — units requested",
  "currency":            "string  — resolved currency (from DB record or RFQ fallback)",
  "listed_unit_price":   "float   — full catalog unit price (set by quote agent)",
  "lead_time_days":      "int     — fulfillment lead time",
  "availability_status": "string  — e.g. 'IN_STOCK', 'LIMITED', 'OUT_OF_STOCK'",
  "minimum_order_qty":   "int     — MOQ from vendor-product record"
}
```

> **Note:** `listed_unit_price` is absent in the RFQ skeleton (before `quote_product`
> runs). Guard against `null` when rendering the quote stage.

---

### 1b. `po` sub-object schema

Written when a valid PO arrives from the buyer.

```json
{
  "po_number":     "string  — e.g. 'PO-20240101-ABCD'",
  "rfq_reference": "string  — must match rfq_id",
  "currency":      "string",
  "line_items": [
    {
      "product_id":   "string",
      "sku":          "string",
      "unit_price":   "float  — agreed negotiated price",
      "quantity":     "int",
      "total_price":  "float"
    }
  ],
  "total_amount":  "float",
  "delivery_date": "string  — ISO-8601 date"
}
```

---

### 1c. `grn` sub-object schema

Written when a valid GRN_CREATED envelope arrives.

```json
{
  "grn_number":  "string  — e.g. 'GRN-20240101-WXYZ'",
  "po_number":   "string  — must match po.po_number",
  "received_at": "string  — ISO-8601 datetime",
  "line_items": [
    {
      "sku":           "string",
      "unit_quantity": "int  — actually received units"
    }
  ]
}
```

## 2. VendorThreadStatus Lifecycle — State Machine

```
[null]
  │
  ▼ [RFQ received]
RFQ_RECEIVED
  │
  ▼ [quote_agent sends QUOTE envelope]
QUOTE_SENT
  │
  ├─ [buyer sends COUNTER_OFFER] ──────────────────────────────┐
  │                                                             ▼
  │                                                NEGOTIATION_IN_PROGRESS ◄──┐
  │                                                             │              │
  │                                                             │ [vendor COUNTER_OFFER]
  │                                                             │              │
  │                                            ┌────────────────┴──────────────┘
  │                                            │
  │                                            ├─ [vendor or buyer ACCEPT] ──► ACCEPTED
  │                                            ├─ [vendor sends WALKAWAY] ───► VENDOR_WALKED_AWAY ──► RFQ_CLOSED
  │                                            └─ [buyer WALKAWAY] ──────────► BUYER_WALKED_AWAY ───► RFQ_CLOSED
  │
  ├─ [buyer ACCEPT] ──────────────────────────────────────────► ACCEPTED
  ├─ [buyer WALKAWAY] ────────────────────────────────────────► BUYER_WALKED_AWAY ──► RFQ_CLOSED
  └─ [buyer RFQ_CLOSED] ──────────────────────────────────────► RFQ_CLOSED (terminal)

ACCEPTED
  ├─ [PO received + validated] ──────────────────────────────► PO_RECEIVED
  └─ [buyer RFQ_CLOSED] ──────────────────────────────────────► RFQ_CLOSED

PO_RECEIVED
  │
  ▼ [acknowledge_po tool sends PO_ACKNOWLEDGED]
PO_ACKNOWLEDGED
  │
  ▼ [GRN_CREATED received + validated]
GRN_RECEIVED
  │
  ▼ [submit_invoice tool sends INVOICE_SUBMITTED]
INVOICE_SUBMITTED
  │
  ▼ [PROCESS_COMPLETE received + validated]
COMPLETE (terminal)

RFQ_CLOSED (terminal)
```

**Trigger summary by message type:**

| Inbound message type | Status transition(s) triggered |
|---|---|
| `RFQ` | `null` → `RFQ_RECEIVED` |
| `COUNTER_OFFER` (buyer) | current → `NEGOTIATION_IN_PROGRESS` |
| `ACCEPT` (buyer) | current → `ACCEPTED` |
| `WALKAWAY` (buyer) | current → `BUYER_WALKED_AWAY` → `RFQ_CLOSED` |
| `RFQ_CLOSED` | current → `RFQ_CLOSED` |
| `PO` | current → `PO_RECEIVED` (after validation) |
| `GRN_CREATED` | current → `GRN_RECEIVED` (after validation) |
| `PROCESS_COMPLETE` | `INVOICE_SUBMITTED` → `COMPLETE` (after validation) |
| Outbound QUOTE (tool) | *(implicit — quote agent runs after RFQ_RECEIVED)* |
| Outbound COUNTER_OFFER/ACCEPT/WALKAWAY (tool) | `ACCEPTED`, `VENDOR_WALKED_AWAY`, etc. via `negotiation/callback.py` |

## 3. Allowed Transition Map

From `_ALLOWED_TRANSITIONS` in `procu_forge_vendor/communication_status.py`.

| From status | Valid next statuses |
|---|---|
| `null` (no status yet) | `RFQ_RECEIVED` |
| `RFQ_RECEIVED` | `QUOTE_SENT` |
| `QUOTE_SENT` | `NEGOTIATION_IN_PROGRESS`, `ACCEPTED`, `BUYER_WALKED_AWAY`, `VENDOR_WALKED_AWAY`, `RFQ_CLOSED` |
| `NEGOTIATION_IN_PROGRESS` | `NEGOTIATION_IN_PROGRESS` (self-loop on each counter), `ACCEPTED`, `BUYER_WALKED_AWAY`, `VENDOR_WALKED_AWAY`, `RFQ_CLOSED` |
| `ACCEPTED` | `PO_RECEIVED`, `RFQ_CLOSED` |
| `BUYER_WALKED_AWAY` | `RFQ_CLOSED` |
| `VENDOR_WALKED_AWAY` | `RFQ_CLOSED` |
| `RFQ_CLOSED` | *(terminal — no further transitions)* |
| `PO_RECEIVED` | `PO_ACKNOWLEDGED` |
| `PO_ACKNOWLEDGED` | `GRN_RECEIVED` |
| `GRN_RECEIVED` | `INVOICE_SUBMITTED` |
| `INVOICE_SUBMITTED` | `COMPLETE` |
| `COMPLETE` | *(terminal — no further transitions)* |

> **Invalid transitions are logged but not rejected** — the system writes the new status
> regardless to avoid permanently wedging a session. Invalid transitions appearing in
> production logs indicate a sequencing bug in the buyer agent.

## 4. Negotiation Price Anchors

Four price fields exist in state, each with a distinct lifecycle. Together they drive the
negotiation price visualisation.

| State key | Meaning | Set when | Display label |
|---|---|---|---|
| `product.listed_unit_price` | Full catalog unit price, the vendor's reference / anchor | `quote/tools.py` after Firestore lookup | "Catalog price" / "List price" |
| `latest_offer_price` | Vendor's most recently offered unit price | `quote/tools.py` (opening = 95% of listed); `send_response` on each vendor COUNTER_OFFER or ACCEPT | "Vendor offer" |
| `latest_buyer_price` | Buyer's most recently proposed unit price | `send_response` when `buyer_proposed_price` is provided (i.e., inbound COUNTER_OFFER or ACCEPT round) | "Buyer offer" |
| `accepted_price` | Final agreed unit price — authoritative | `negotiation/callback.py` when the vendor sends an outbound ACCEPT envelope | "Agreed price" |

**Rendering a negotiation price bar:**

```
[last_selling_price]──────[latest_buyer_price]──[latest_offer_price]──[listed_unit_price]
      floor (hidden)            buyer anchor          vendor anchor         catalog anchor
```

- `last_selling_price` is the server-enforced floor (90% of `listed_unit_price`). It is
  **never surfaced to the buyer** but can be shown in the vendor-side dashboard to
  indicate the minimum the vendor will accept.
- Once `accepted_price` is set, replace both offer bars with a single "Agreed" marker.
- `listed_unit_price` can be `null` until `quote_product` runs — guard rendering.
- The opening quote is always `listed_unit_price × 0.95` (5% opening discount).

## 5. Message Log Structure (`communication` key)

`state["communication"]` is a `list[dict]`. Each element is a complete A2A envelope.

### 5a. Common envelope fields

Every envelope shares this outer skeleton:

```json
{
  "message_id":   "string  — UUID, unique per message; used for dedup",
  "rfq_id":       "string  — thread identifier",
  "vendor_id":    "string",
  "from_agent":   "string  — 'buyer_agent' | 'vendor_agent'",
  "to_agent":     "string  — 'vendor_agent' | 'buyer_agent'",
  "message_type": "string  — see types below",
  "round":        "int     — negotiation round (0 for RFQ/QUOTE)",
  "timestamp":    "string  — ISO-8601 UTC",
  "payload":      "dict    — type-specific content"
}
```

**Direction:** `from_agent == "buyer_agent"` → inbound to vendor. `from_agent == "vendor_agent"` → outbound from vendor.

---

### 5b. Message types and payload shapes

**`RFQ`** — inbound, round 0
```json
{
  "payload": {
    "item": { "product_id": "...", "sku": "...", "quantity": 10, "unit": "each" },
    "required_by_date": "2024-03-15",
    "response_deadline": "2024-01-16T12:00:00Z"
  }
}
```

**`QUOTE`** — outbound, round 0
```json
{
  "payload": {
    "unit_price":        95.00,
    "currency":          "USD",
    "quantity":          10,
    "total_amount":      950.00,
    "lead_time_days":    5,
    "response_deadline": "2024-01-17T12:00:00Z",
    "valid_until":       "2024-01-23T12:00:00Z"
  }
}
```

**`COUNTER_OFFER`** — buyer or vendor, round ≥ 1
```json
{
  "payload": {
    "unit_price":        91.00,
    "currency":          "USD",
    "is_final":          false,
    "response_deadline": "2024-01-18T12:00:00Z"
  }
}
```
`is_final: true` means the vendor will not counter again — display "Best & Final" badge.

**`ACCEPT`** — buyer or vendor
```json
{
  "payload": {
    "unit_price": 92.50,
    "currency":   "USD"
  }
}
```

**`WALKAWAY`** — buyer or vendor
```json
{
  "payload": {
    "reason":          "PRICE_TOO_LOW",
    "last_unit_price": 91.00
  }
}
```

**`RFQ_CLOSED`** — inbound from buyer (losing vendors)
```json
{ "payload": {} }
```

**`PO`** — inbound, purchase phase
```json
{
  "payload": {
    "po_number":     "PO-20240101-ABCD",
    "rfq_reference": "...",
    "currency":      "USD",
    "line_items":    [{ "product_id": "...", "sku": "...", "unit_price": 92.50, "quantity": 10, "total_price": 925.00 }],
    "total_amount":  925.00,
    "delivery_date": "2024-03-15"
  }
}
```

**`PO_ACKNOWLEDGED`** — outbound
```json
{ "payload": { "po_number": "PO-20240101-ABCD" } }
```

**`GRN_CREATED`** — inbound
```json
{
  "payload": {
    "grn_number":  "GRN-20240101-WXYZ",
    "po_number":   "PO-20240101-ABCD",
    "received_at": "2024-03-15T09:30:00Z",
    "line_items":  [{ "sku": "...", "unit_quantity": 10 }]
  }
}
```

**`INVOICE_SUBMITTED`** — outbound
```json
{
  "payload": {
    "invoice_number": "INV-<rfq_id>-<random8>",
    "po_number":      "PO-20240101-ABCD",
    "invoice_date":   "2024-03-15",
    "due_date":       "2024-04-14",
    "grn_reference":  "GRN-20240101-WXYZ",
    "line_items":     [{ "sku": "...", "quantity": 10, "unit_price": 92.50, "total_price": 925.00 }],
    "total_amount":   925.00
  }
}
```

**`PROCESS_COMPLETE`** — inbound, terminal
```json
{
  "payload": {
    "po_number":      "PO-20240101-ABCD",
    "grn_number":     "GRN-20240101-WXYZ",
    "invoice_number": "INV-<rfq_id>-<random8>"
  }
}
```

### 5c. Rendering the communication timeline

1. Iterate `state["communication"]` in order — it is already chronological.
2. Group by `round` for a round-by-round accordion view.
3. Determine direction from `from_agent`: `"buyer_agent"` = left/inbound; `"vendor_agent"` = right/outbound.
4. Use `message_type` to pick the card template (price card, document card, status pill).
5. The first entry is always the RFQ (round 0, `from_agent == "buyer_agent"`).
6. QUOTE immediately follows as the vendor's round-0 reply.

## 6. Frontend Status Categories

### Display guide

| Category | Statuses | Suggested UI treatment |
|---|---|---|
| **Incoming** | `RFQ_RECEIVED` | Amber badge "New RFQ"; show product card with quantity and required_by_date |
| **Quoting** | `QUOTE_SENT` | Blue badge "Quoted"; show quote price, valid_until, lead_time_days |
| **Negotiating** | `NEGOTIATION_IN_PROGRESS` | Live indicator with round counter; show both buyer and vendor offer bars; show `is_final` badge when `vendor_is_final == true` |
| **Deal reached** | `ACCEPTED` | Green badge "Accepted"; show `accepted_price`, awaiting PO |
| **No deal — buyer** | `BUYER_WALKED_AWAY` | Muted grey "Buyer walked away"; show walkaway reason from last WALKAWAY envelope |
| **No deal — vendor** | `VENDOR_WALKED_AWAY` | Muted grey "Vendor walked away" |
| **Closed** | `RFQ_CLOSED` | Strikethrough / archived state; thread is over |
| **PO pending** | `PO_RECEIVED` | Yellow "PO Received"; show PO number, line items, delivery date; action: acknowledge |
| **PO acknowledged** | `PO_ACKNOWLEDGED` | Blue "PO Acknowledged"; awaiting goods dispatch / GRN |
| **GRN received** | `GRN_RECEIVED` | Blue "GRN Received"; show received quantities; action: submit invoice |
| **Invoice sent** | `INVOICE_SUBMITTED` | Amber "Invoice Submitted"; show invoice number, due date, total amount |
| **Complete** | `COMPLETE` | Green banner "Transaction Complete"; show all reference numbers |

| `RFQ_RECEIVED` | partial (no price) | — | — | — | — | — |

## 7. Validation Rules (frontend awareness)

The server enforces these rules and returns `{"ok": false, "error": "...", "details": [...]}` 
when violated. The frontend should surface the `details` array to the vendor operator.

| Inbound message | Validation performed |
|---|---|
| `PO` | `rfq_reference` must match `rfq_id`; `currency` must match negotiated currency; `line_items[].unit_price` must match `accepted_price` (±0.01); `total_amount` must equal sum of `line_items[].total_price` (±0.01) |
| `GRN_CREATED` | `po_number` must match `po.po_number`; all GRN `sku` values must exist in PO `line_items` |
| `PROCESS_COMPLETE` | Must arrive while status is `INVOICE_SUBMITTED`; `po_number`, `grn_number`, `invoice_number` all required and must match stored values |
| Any message | `message_id` dedup — duplicate envelopes are silently acknowledged without re-processing |
| `RFQ`, `COUNTER_OFFER`, `ACCEPT`, `WALKAWAY` | `payload.response_deadline` (ISO-8601) must not be in the past |
